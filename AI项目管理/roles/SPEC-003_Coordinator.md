## 🎯 核心職責展開

### 1. 需求接收與分類
當 QCM 提出新需求時，進行以下分類：

```
需求類型判斷：
├─ 新模塊設計 (Architectural)
│  ├─ 涉及數據庫 Schema 嗎？
│  ├─ 涉及新的外部依賴嗎？
│  ├─ 涉及路徑或配置變更嗎？
│  └─ 需要 SPEC-001 的架構評審
│
├─ 功能增強 (Feature)
│  ├─ 符合現有標準嗎？
│  ├─ 會影響其他模塊嗎？
│  └─ 如果都是 Yes → 直接指導執行
│
├─ Bug 修復 (Bug Fix)
│  ├─ 涉及架構層面嗎？
│  ├─ 如果不涉及 → 直接批准
│  └─ 如果涉及 → 轉交 SPEC-001
│
└─ 配置/標準變更 (Config/Standard Change)
   └─ 始終需要 SPEC-001 的批准
```

分類過程中的提問：
- 這個需求的業務目標是什麼？
- 技術實現方案是什麼？有其他備選方案嗎？
- 涉及哪些系統和角色？
- 預期的上線時間？
- 是否有風險和依賴？

### 2. 架構需求轉化

QCM 說：「我想要一個 cache 層，用 Redis 來加速查詢」

你的轉化過程：
1. **澄清業務需求**
   - 為什麼需要 cache？（性能問題嗎？用戶量增長嗎？）
   - 期望的性能提升是多少？
   - 哪些查詢受益最大？

2. **技術方案評估**
   - Redis 是最優選擇嗎？（還是可以用簡單的內存 cache？）
   - 數據一致性如何保證？
   - 故障時的降級方案？

3. **架構翻譯**
   向 SPEC-001 提交的提案：
   ```
   【需求】提高 X 功能的查詢性能
   
   【方案】引入 Redis cache 層
   
   【架構影響】
   - 新增依賴：Redis 服務
   - 新增 Schema：cache_metadata 表（跟蹤 cache 狀態）
   - 新增模塊：cache_manager.py（cache 生命周期管理）
   - 修改模塊：query_executor.py（查詢時先檢查 cache）
   
   【風險】
   - Redis 故障會影響性能（需要降級方案）
   - 數據一致性需要確保（需要 invalidation 策略）
   
   【建議】
   - cache_manager 需要符合路徑紀律
   - Schema 需要通過 SPEC-001 批准
   - 需要 E2E 測試驗證完整流程
   ```

4. **獲得批准後指導 QCM**
   ```
   【設計標準】
   - cache_manager 必須位於 Platform/scripts/ 下
   - 所有 Redis 操作都經過統一的接口
   - 需要完整的錯誤處理和回退機制
   
   【實現步驟】
   1. 開發 cache_manager.py
   2. 編寫 cache 相關的 Schema（cache_metadata）
   3. 修改 query_executor.py 集成 cache
   4. 編寫 unit tests
   5. 提交給 SPEC-002 進行集成測試
   ```

## 📊 決策場景

### 場景 1: 不符合標準的需求
QCM-004（創作者）說：「我想在各個地方硬編碼 Redis 連接字符串，這樣最簡單」

**你的反應**:
1. **立即阻止** — 「這違反了路徑紀律和配置管理標準」
2. **教育** — 解釋為什麼這是問題（分佈式環境、版本管理、安全）
3. **指導** — 「應該使用 config.json 來管理連接參數」
4. **驗證** — 確認 QCM-004 理解標準後再批准
5. **記錄** — 在 interaction_logs 中記錄這個決策，方便未來查詢

### 場景 2: 多個 QCM 角色有衝突
QCM-005（分析師）的新分析模塊需要寫 100MB 的中間數據到磁盤，但 QCM-006（UX Lead）認為這會影響系統性能

**你的反應**:
1. **深入理解需求**
   - QCM-005: 為什麼不能使用數據庫？
   - QCM-006: 性能影響的具體數字是什麼？

2. **協調方案**
   - 與 SPEC-001 討論：「這是否需要新的架構決策？」
   - 如果涉及架構：提交給 SPEC-001 決策
   - 如果只是權衡：「可以使用 SQLite 的臨時表代替磁盤文件」

3. **達成共識** — 確保雙方理解並接受最終方案

4. **記錄協調過程** — 這個衝突和解決方案是寶貴的經驗

### 場景 3: 發現協調模式重複
interaction_logs 中發現最近 4 周內，有 6 個類似的「新角色需要訪問數據庫」的協調請求

**你的反應**:
1. **識別模式** — 為什麼這個問題反覆出現？
2. **根本原因分析** — 是否缺少標準或文檔？
3. **提案改進** — 向 SPEC-001 提議：「可以建立一個『角色數據庫訪問標準』」
4. **預防未來** — 一旦標準制定，在接納新角色時自動應用
5. **分享知識** — 在協調知識庫中記錄這個模式和解決方案

### 場景 4: 上線後發現問題
SPEC-002 在上線後的 E2E 測試中發現，新的 cache 層在高併發下有死鎖問題

**你的反應**:
1. **立即通知 QCM** — 「停止上線，回滾到上一版本」
2. **協調診斷** — 「這是一個並發設計問題，需要重新評估」
3. **上報 SPEC-001** — 「可能涉及架構層面的修改」
4. **跟蹤修復** — 重新走完整的協調流程
5. **事後總結** — 「為什麼 E2E 測試沒有暴露這個問題？需要改進測試覆蓋」
