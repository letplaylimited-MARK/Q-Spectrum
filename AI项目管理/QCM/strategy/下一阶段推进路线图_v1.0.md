# QCM × 幽灵通道 下一阶段推进路线图

**版本**: v1.0-Roadmap  
**日期**: 2026-04-05  
**策略**: 工程主线 + 产品/商业主线双线并行  
**目标**: 从“完整成果体系”推进到“可持续演化的平台能力与对外试点能力”  

---

## 一、路线图总览

当前阶段，QCM 与幽灵通道已经形成了完整的理论、协议、工程与文档资产闭环：

- 主论文终稿与投稿版
- 幽灵通道独立论文与投稿版
- 技术白皮书与高管版白皮书
- 协议规范稿 / RFC 草案 / RFC 可发布版
- Python SDK MVP / TypeScript SDK MVP
- schemas / examples / mapping / validator / tests / CI / CLI
- PoC Phase 1-4
- 自用归档版总目录

因此，下一阶段不再是“补文档”，而是要完成从**成果沉淀**到**持续推进机制**的切换。

本路线图采用双线并行结构：

1. **工程主线**：继续强化协议、SDK、验证与开源准备
2. **产品/商业主线**：验证应用场景、形成试点、建立价值闭环

---

## 二、阶段划分

### Phase A：工程收敛期（0-4 周）

目标：把当前成果从“高质量 MVP”收敛为“可稳定演示、可继续开发”的工程基座。

### Phase B：试点验证期（1-3 个月）

目标：围绕 1-2 个高价值场景做真实试点，验证技术与商业价值。

### Phase C：标准化与开源期（3-6 个月）

目标：推进 RFC / SDK / 开源结构，把幽灵通道从内部能力升级为外部可采用资产。

### Phase D：平台扩展期（6-12 个月）

目标：把幽灵通道从“协议能力”扩展到“平台基础设施”，带动 QCM 整体演化。

---

## 三、工程主线

## Phase A：工程收敛期（0-4 周）

### A1. Python SDK 收敛

**目标**：把 Python SDK 从 MVP 推进到“稳定工程原型”。

**工作项**：
1. 将 `EncryptedStream` 的完整对象流进一步覆盖到更多路径
2. 将 `SyncResult / AckMessage / ErrorObject` 的使用从“最小必需”扩展到全流程
3. 引入更严格的对象级 schema 校验策略
4. 完善 CLI / demo runner 输出格式

**验收标准**：
- Python SDK 测试数 ≥60
- 所有核心路径均通过 schema 校验
- CLI 能稳定演示 Memory / Workflow 两条主路径

### A2. TypeScript SDK 追平

**目标**：把 TypeScript SDK 追到与 Python 接近的协议语义水平。

**工作项**：
1. 接入更完整的对象流
2. 补齐 workflow path 的 delta / snapshot / encrypted stream 细节
3. 强化 ACK / replay / completionMode 的等待与终态逻辑
4. 增加 TypeScript 侧的 schema 校验接入

**验收标准**：
- TypeScript 测试数 ≥20
- Memory / Workflow 双路径都具备对象流和 ACK 终态等待

### A3. 资产链收敛

**目标**：保证 `schema → example → mapping → validator → CI` 五层完全闭环。

**工作项**：
1. 清理残余路径或命名不一致
2. 增加更多 example fixture
3. 明确 schema registry 的维护规则

**验收标准**：
- validator 全绿
- pre-commit / CI 全绿
- example-schema-map 无孤立项

---

## Phase B：试点验证期（1-3 个月）

### B1. 选择 1-2 个高价值试点场景

优先推荐：

| 场景 | 原因 |
|------|------|
| 多智能体记忆同步 | 最能体现幽灵通道的“通信是根基” |
| 因果工作流引擎 | 最能体现恢复、自愈、可追溯能力 |

### B2. 试点目标

**技术指标**：
- 带宽降低 ≥80%
- ACK / replay / 恢复逻辑在真实场景下稳定
- 角色 / 工作流状态一致性 ≥99%

**业务指标**：
- 决策耗时下降
- 人工排障时间下降
- 试点用户满意度可量化

### B3. 试点交付物

1. PoC 试点部署记录
2. 技术指标报告
3. 场景价值复盘
4. 下一步投入建议

---

## Phase C：标准化与开源期（3-6 个月）

### C1. RFC 收敛

**目标**：把 RFC 草案推进到更接近正式标准评审状态。

**工作项**：
1. 再做一轮严格协议审校
2. 明确 MVP Required / Optional / Experimental 边界
3. 形成更清晰的版本策略

### C2. 开源发布准备

**目标**：将幽灵通道工程资产整理成可公开发布结构。

**最小发布清单**：
- LICENSE
- CONTRIBUTING
- CHANGELOG
- SDK 目录
- schemas/examples
- Developer Guide
- PoC 示例

### C3. 开发者入口

**目标**：让外部开发者可以在最短时间跑通第一个示例。

**交付物**：
1. Quick Start
2. 最小示例
3. FAQ / 常见错误

---

## Phase D：平台扩展期（6-12 个月）

### D1. 幽灵通道从协议走向平台能力

目标：让幽灵通道成为 QCM 中独立的基础设施层，而不是只服务于单一 PoC。

### D2. 技术扩展方向

优先方向：
1. 多模态协议能力继续深化
2. 后量子安全优化
3. 可验证计算 / 零知识证明
4. 联邦学习协调能力

### D3. QCM 主体系协同演进

把幽灵通道持续回写到：
- 主论文
- 白皮书
- 规范稿
- SDK
- 场景试点报告

形成一个持续闭环，而不是一次性成果。

## 四、产品 / 商业主线

## Phase A：价值叙事收敛（0-4 周）

### A1. 对外讲法统一

要形成三种统一叙事：

| 读者 | 文档 |
|------|------|
| 研究者 | 独立论文稿 |
| CTO / 架构师 | 技术白皮书 |
| 高管 / 投资人 | 高管版白皮书 |

### A2. 价值主张固定

建议固定为：

> 幽灵通道不是同步工具，而是 AI 协作系统的通信基础设施层。

---

## Phase B：场景验证（1-3 个月）

### B1. 建立应用场景矩阵

建议场景：
- 多智能体记忆同步
- 因果工作流
- 高合规行业协作
- 多模态协作平台

### B2. 建立价值量化模型

至少要沉淀：
- 成本节省
- 带宽降低
- 恢复时间
- 协作效率
- 审计价值

---

## Phase C：对外试点与合作（3-6 个月）

目标：
- 找到 1-2 个真实试点对象
- 用白皮书 / 高管版白皮书支撑沟通
- 用 SDK / PoC 支撑技术验证

### 最佳合作对象类型

1. 有多 Agent / 多工作流协作需求的团队
2. 对数据一致性敏感的团队
3. 对合规和审计有明确要求的团队

---

## 五、双线协同关系

工程主线和产品主线不是平行独立，而是相互驱动：

| 工程推进 | 反哺商业 |
|---------|---------|
| SDK 更稳定 | 白皮书更可信 |
| PoC 数据更强 | ROI 叙事更具体 |
| RFC 更清晰 | 对外合作更容易 |

| 产品反馈 | 反哺工程 |
|---------|---------|
| 真实场景问题 | 协议边界更清晰 |
| 试点阻力 | SDK/CLI/手册更完善 |
| 高管关注点 | 白皮书与 RFC 表述更聚焦 |

## 八、最终建议

当前阶段最重要的不是再无限扩写材料，而是：

> **把已有的完整成果体系，转化成“可持续推进”的项目机制。**

因此，建议后续按以下原则推进：

1. 工程先收敛，产品再外推
2. 每一轮新成果都回写文档体系
3. 让幽灵通道优先成为 QCM 的独立输出能力
