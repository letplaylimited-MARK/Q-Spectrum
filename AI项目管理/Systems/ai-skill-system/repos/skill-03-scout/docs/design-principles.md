# 设计原则 · Design Principles
## 开源技能侦察官 v1.0.1

---

## 核心设计哲学

### 原则一：诚实优于完整

宁愿声明「未找到满足需求的开源方案」，也不勉强推荐质量不足的方案。
gaps_found 机制的存在正是为了让用户收到诚实的评估，而非被虚假推荐误导。

### 原则二：幻觉防护是模板级责任，不是 AI 自觉

H1/H2/H3 幻觉防护规则必须嵌入到每个输出模板中，作为固定字段存在。
不依赖 AI 在每次执行时「记得」标注，而是通过格式强制保证。

### 原则三：交接包降低用户搬运成本

Skill 间协作的本质是用户搬运结构化中间产物。
交接包的设计目标是：用户只需「复制→粘贴→发送」三步，零理解负担。

### 原则四：边界清晰，不越权

开源技能侦察官的职责止于「选型推荐」。
工程化打包（Skill 02）、提示词优化（Skill 01）、操作手册生成（Skill 04）、验收测试（Skill 05）均不在本 Skill 职责范围内。
遇到越权请求时，主动声明「此需求建议转介给 Skill XX」。

### 原则五：飞轮贡献（v1.0.1 新增）

Skill 03 的评估数据是体系飞轮的重要驱动力：
- **gaps_found 数据**：反馈给体系维护者，识别哪些领域开源生态薄弱
- **维度评分模式**：高频低分维度提示 Skill 02 需要重点关注的工程质量方向
- **技术栈分布**：帮助体系了解用户最常用的技术栈，优化 Sub-Skill 02 的搜索策略

## IOSE 原则应用

| 原则 | 在本 Skill 中的体现 |
|---|---|
| Independent（独立）| 可单独使用，不依赖其他 Skill 的预执行 |
| One-responsibility（单一职责）| 仅负责开源技能搜索与评估，不跨越边界 |
| Structured output（结构化输出）| 所有输出均有固定格式（分析表/评估矩阵/交接包）|
| Explicit trigger（明确触发）| 触发词专属于开源技能领域，避免泛化误触发 |

## PAST 原则应用（激活卡片）

| 原则 | 验证 |
|---|---|
| Paste-ready（即粘即用）| 激活卡片可直接粘贴到任意 AI 对话，无需修改 |
| Activation signal（激活信号）| 明确触发词列表，AI 收到后立即输出确认 |
| Self-contained（自给自足）| 激活卡片包含完整的工作模式描述，无需额外说明 |
| Trigger mapping（触发映射）| 触发词与能力一一对应，无歧义 |

## NAD 原则应用（幻觉防护）

| 原则 | 在本 Skill 中的体现 |
|---|---|
| No fabrication（不虚构）| Star 数/更新日期/版本号 一律标注 [需用户核实]，绝不填写具体数字 |
| Admit uncertainty（承认不确定）| gaps_found 机制直接声明无法找到满足方案，不强行推荐 |
| Defer to user（让用户决策）| 评估结果提供信息和推荐理由，最终决策权归用户 |
