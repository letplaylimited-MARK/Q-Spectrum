# Sub-Skill 02 · 开源搜索 Open Source Search
## 阶段 02/4

---

## 职责声明

| 项目 | 内容 |
|---|---|
| 输入 | 阶段01 确认后的技能需求清单（REQ-XXX 格式）|
| 输出 | 每个需求 2-3 个备选开源方案（含开源协议/功能描述）|
| 门控条件 | 自动进入阶段03（无需用户门控）|
| 禁止行为 | 不虚构 Star 数/更新日期/版本号；不推荐私有或非开源方案 |

---

## 执行逻辑

### Step 1：遍历需求清单

对每个 REQ-XXX（类别非 redirect_s01/s02 的）依次执行搜索。

跳过类别为 redirect_s01 / redirect_s02 的需求，在输出中注明「此需求已标注转介，不执行开源搜索」。

### Step 2：按类别选择搜索策略

| 需求类别 | 优先搜索范围 | 常见候选库方向 |
|---|---|---|
| nlp | PyPI / HuggingFace / GitHub | transformers / spacy / nltk / snownlp / jieba |
| vision | PyPI / GitHub | opencv / pillow / torchvision / paddleocr |
| data | PyPI / GitHub | pandas / polars / matplotlib / seaborn / plotly |
| api | PyPI / npm / GitHub | requests / httpx / fastapi / flask / express |
| ml | PyPI / GitHub | scikit-learn / pytorch / tensorflow / xgboost |
| scraping | PyPI / GitHub | scrapy / playwright / selenium / beautifulsoup4 |
| storage | PyPI / GitHub | sqlalchemy / pymongo / redis-py / tinydb |

### Step 3：输出每个需求的备选方案

每个需求输出 2-3 个方案，严格遵循幻觉防护三级规则：

**H1规则（输出模板中强制嵌入）**：
- Star 数字段：必须写 `[需用户核实]`，不写具体数字
- 更新日期字段：必须写 `[需用户核实]`

**H2规则**：
- 维护状态：如无法确认，写 `[需用户核实是否仍在维护]`

**H3规则**：
- 版本号：在安装命令后附注 `# 版本号需用户核实，建议安装最新稳定版`
- 依赖兼容：标注 `# 请验证与当前环境的兼容性`

### Step 4：处理 gaps_found（找不到满足方案时）

触发条件：
- 需求高度特定于某行业领域（如医疗专用术语分析）
- 需求结合多个罕见功能，无现成开源库覆盖
- 已知相关领域没有成熟开源项目

处理方式：
```
REQ-00X：[需求名称] — 未找到满足需求的成熟开源方案
原因：[具体说明]
建议路径（三选一）：
A. 降低需求标准：使用 [已知最接近方案] 部分满足，补充 [X] 的缺失
B. 定制开发：调用 Skill 02（SOP工程师）从零构建专用 Skill
C. 付费替代：考虑使用 [已知付费 API/SaaS 服务名称] 代替开源方案
```

## 防错护栏

| 场景 | 处理方式 |
|---|---|
| 某需求只找到1个方案 | 声明「仅找到1个符合要求的方案，已列出」，不强行凑数 |
| 某需求找到超过3个方案 | 选取综合质量最高的3个，其余可在阶段03中注明「另有备选」|
| 用户要求推荐付费/私有方案 | 声明「本侦察官专注开源方案，付费方案建议在 gaps_found 路径C中考虑」|
| 需求类别为 redirect_s01/s02 | 在输出中注明「此需求已标注转介至 Skill 0X，跳过开源搜索」|
| 安装命令不确定 | 写 `pip install [包名]  # 请在官方文档确认正确命令` |
