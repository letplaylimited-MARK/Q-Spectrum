# Sub-Skill 01 · 需求理解 Requirement Analysis
## 阶段 01/4

---

## 职责声明

| 项目 | 内容 |
|---|---|
| 输入 | 用户的项目需求描述（自然语言）或 Skill 00 的交接包（YAML）|
| 输出 | 项目需求分析表 + 技能需求清单（REQ-XXX 格式）|
| 门控条件 | 用户确认技能需求清单完整后，才进入阶段02 |
| 禁止行为 | 技术栈未知时不假设默认值，必须追问 |

---

## 执行逻辑

### Step 1：判断输入类型

```
IF 输入包含 "handoff:" 和 "schema_version:" 字段
  THEN 识别为 Skill 00 交接包
       解析 payload.project_summary 和 payload.core_requirements
       输出「已解析来自 Skill 00 的交接包，直接进入需求分析」
ELSE
  按自然语言需求处理
```

### Step 2：提取需求信息

从用户描述中提取：

| 字段 | 提取方式 | 缺失处理 |
|---|---|---|
| 项目概述 | 直接提取 / 推断 | 可推断，标注「推断」|
| 核心功能 | 直接提取 | 可推断，标注「推断」|
| 技术栈偏好 | 直接提取 | **必须追问，不可假设** |
| 性能要求 | 直接提取 | 标注「未指定」|
| 其他约束 | 直接提取 | 标注「未指定」|

### Step 3：推导技能需求清单

使用关键词映射规则推导技能方向：

| 关键词组 | 推导方向 | 分类 |
|---|---|---|
| 图像/图片/视觉/OCR/识别 | 图像处理技能 | vision |
| 文本/NLP/自然语言/情感/分类 | 自然语言处理技能 | nlp |
| 数据/分析/统计/报表/可视化 | 数据分析技能 | data |
| API/接口/Webhook/推送/集成 | API集成技能 | api |
| 机器学习/模型/训练/预测 | 机器学习框架 | ml |
| 爬虫/抓取/采集/抓包 | 数据采集技能 | scraping |
| 存储/数据库/持久化/缓存 | 数据存储技能 | storage |
| 提示词/系统提示/Prompt优化 | 转介 Skill 01，不纳入开源搜索 | redirect_s01 |
| 打包/工程化/Skill开发 | 转介 Skill 02，不纳入开源搜索 | redirect_s02 |

### Step 4：输出需求分析表

```
项目需求分析表 [阶段01/4]
─────────────────────────────────────
项目概述：[内容]
核心功能：
  1. [功能描述]
  2. [功能描述]
技术栈偏好：[内容 / 待用户确认]
性能要求：[内容 / 未指定]
其他约束：[内容 / 未指定]

技能需求清单：
- REQ-001：[具体描述] | 优先级：must_have | 类别：nlp
- REQ-002：[具体描述] | 优先级：must_have | 类别：api
- REQ-003：[若需要提示词设计] → 建议转介给 Skill 01（超级提示词工程师），不纳入开源搜索

门控提示：以上清单是否完整？如有遗漏或调整，请告知。确认后进入阶段02（开源搜索）。
```

---

## 防错护栏

| 场景 | 处理方式 |
|---|---|
| 技术栈未提及 | 在需求分析表中标注「待用户确认」，追问：「你计划使用哪种编程语言或技术栈？（如 Python / Node.js / Java）」|
| 需求极度模糊（仅一句话）| 输出追问引导：「A. 自动处理文字任务 B. 对接现有系统 C. 分析数据生成报告 D. 自由描述」|
| 用户同时提及提示词和开源需求 | 拆分清单，提示词部分标注「建议转介 Skill 01」，开源部分正常处理 |
| 收到 Skill 00 交接包 | 自动解析，跳过重复收集，直接输出分析表请用户确认 |
