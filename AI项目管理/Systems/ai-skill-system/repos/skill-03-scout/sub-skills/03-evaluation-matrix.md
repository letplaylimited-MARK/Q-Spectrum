# Sub-Skill 03 · 七维度评估 Seven-Dimension Evaluation
## 阶段 03/4

| 项目 | 内容 |

| 输入 | 阶段02 产出的备选方案列表 |

## 七维度评估规则

### 权重定义

| 维度 | 权重 | 评分侧重 |
|---|---|---|
| 功能性 Functionality | 30% | 是否完全覆盖 REQ 描述的功能需求；额外功能不加分，功能缺失扣分 |
| 易用性 Usability | 20% | 安装步骤数量、API 调用复杂度、配置文件是否必须 |
| 性能 Performance | 15% | 处理速度、内存占用、是否支持批量/异步处理 |
| 可维护性 Maintainability | 10% | 代码结构清晰度、是否支持二次开发、依赖数量 |
| 社区活跃度 Community | 10% | 更新频率、Issue响应、PR合并活跃度（**评分时标注[需用户核实]**）|
| 兼容性 Compatibility | 10% | 与用户技术栈（语言版本/OS/框架）的匹配程度 |
| 文档完整性 Documentation | 5% | README 质量、是否有使用示例、API 文档完整性 |

### 评分标准（1-5分）

| 分值 | 含义 |
|---|---|
| 5 | 完全满足/极优秀 |
| 4 | 基本满足/良好，有小瑕疵 |
| 3 | 部分满足/一般，需要额外工作 |
| 2 | 勉强满足/较差，有明显缺陷 |
| 1 | 不满足/很差，需要大量改造 |

### 综合得分计算

综合得分 = 功能×0.30 + 易用×0.20 + 性能×0.15 + 可维护×0.10 + 社区×0.10 + 兼容×0.10 + 文档×0.05

---

## 推荐逻辑

```
IF 最高综合得分 >= 3.5
  THEN 推荐得分最高方案
       若同分，优先「功能性」维度得分更高者
       若仍同分，优先「易用性」维度得分更高者

ELSE IF 最高综合得分 >= 2.5
  THEN 推荐最高分方案，但标注「方案质量中等，建议评估是否考虑定制开发」

ELSE（最高分 < 2.5）
  THEN 声明「未找到满足质量标准的方案」
       激活 gaps_found 处理（见 Sub-Skill 02）
       建议转介 Skill 02
```

---

## 执行逻辑

### Step 1：逐需求逐方案打分

对每个 REQ 的每个备选方案，按七维度逐一评分并提供简短说明。

**社区活跃度维度特殊处理**：
- 评分时写 `[需用户核实] 基于已知信息推断：X/5`
- 告知用户核实方式：前往 GitHub 仓库查看 Insights → Contributors / Commits

### Step 2：计算综合得分

用 Python 精度计算（避免浮点偏差）：
综合得分 = round(func*0.30 + ease*0.20 + perf*0.15 + maint*0.10 + comm*0.10 + compat*0.10 + doc*0.05, 2)

### Step 3：输出比较表 + 推荐

每个需求输出：
1. 各方案对比表（一行一方案）
2. 详细评估卡（每个方案的七维分析）
3. 推荐选择 + 推荐理由（3条）+ 风险提示（1-2条）

## 防错护栏

| 场景 | 处理方式 |
|---|---|
| 所有方案综合得分 < 2.5 | 激活 gaps_found，不强行推荐 |
| 用户不同意推荐结果 | 询问具体顾虑，调整评分权重或重新推荐 |
| 用户要求忽略某维度 | 可接受，重新计算并声明「已忽略 X 维度，调整后得分为 Y」|
| 技术栈变更导致兼容性分数变化 | 重新计算兼容维度，输出更新后的推荐 |
