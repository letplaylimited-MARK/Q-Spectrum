# Sub-Skill 04 · 模拟测试 Simulation Test
## Skill Dev SOP — 阶段04 · v1.0.0

---

## 职责声明 | Responsibility

本子Skill对 v1.0.0 工程包执行**三角色虚拟用户测试 + 递归自举测试**，产出完整的缺陷清单（P0/P1/P2分级），为阶段05自主迭代提供精确修复目标。

**输入**：v1.0.0 完整工程包
**输出**：结构化缺陷清单（P0必须修复/P1本版本修复/P2下版本计划）
**门控**：自动进入阶段05（无需用户确认，缺陷清单即触发条件）

---

## 执行逻辑 | Execution Logic

### Step 1 · 测试框架设计

本阶段使用**自举式虚拟测试（Bootstrapping Simulation）**：由AI扮演不同类型的用户，对Skill执行真实的使用场景，发现实际问题。

测试包含4类场景：

| 测试类型 | 执行者 | 场景描述 | 重点检测 |
|---|---|---|---|
| 角色A：绝对新手 | AI扮演 | 第一次接触Skill，完全不了解流程 | 引导清晰度、上手体验 |
| 角色B：进阶用户 | AI扮演 | 有AI工具使用经验，想快速上手 | 效率路径、功能深度 |
| 角色C：边缘案例用户 | AI扮演 | 输入奇怪/不完整/矛盾的需求 | 防错护栏、错误恢复 |
| 递归自举测试 | AI扮演 | 用该Skill来开发它自身 | 逻辑自洽性、循环问题 |

---

### Step 2 · 角色A测试（绝对新手）

**用户画像**：不了解AI Skill、不懂提示词工程、第一次使用

**模拟对话流程：**

```
用户输入（新手）：「我听说可以把文档做成AI技能，但我完全不懂，能帮我吗？」

期望Skill响应：
  □ 识别触发词「把文档做成AI技能」
  □ 输出激活确认（包含引导菜单）
  □ 使用非技术语言，避免行话
  □ 提供A/B/C/D选项引导起点选择

用户输入（新手）：「我选A，我有一个手册文档」

期望Skill响应：
  □ 进入阶段01价值深挖
  □ 请求查看文档内容
  □ 不假设文档内容

用户输入（新手）：「这是我的文档：[粘贴一段混乱的文本]」

期望Skill响应：
  □ 正确解析文档
  □ 输出结构化价值地图
  □ 语言亲和，不使用技术术语
```

**检测重点**：
- [ ] 激活响应是否清晰易懂？
- [ ] 菜单选项是否直观？
- [ ] 全程是否有行话/术语阻碍理解？
- [ ] 阶段进展是否有清晰提示？

---

### Step 3 · 角色B测试（进阶用户）

**用户画像**：熟悉AI工具，想快速完成Skill开发，不想被冗长引导拖慢

**模拟对话流程：**

```
用户输入（进阶）：「开发Skill，我有完整的设计文档，直接从阶段03开始」

期望Skill响应：
  □ 识别「从阶段03开始」的跳跃请求
  □ 建议先完成01-02确保约束清晰（但不强制拦截）
  □ 或允许跳跃并提示风险

用户输入（进阶）：「[提供详细的功能规格文档]，帮我生成SKILL.md」

期望Skill响应：
  □ 快速提取关键信息（无需从头引导）
  □ 直接进入工程构建模式
  □ 生成符合规范的SKILL.md

用户输入（进阶）：「description太长了，帮我精简到1024字符以内」

期望Skill响应：
  □ 精确计算当前字符数
  □ 给出精简版本
  □ 验证精简后仍然完整覆盖核心信息
```

**检测重点**：
- [ ] 是否支持灵活的进入点（不强制从头开始）？
- [ ] 对有经验的用户是否减少了不必要的引导？
- [ ] description字符数计算是否精确？
- [ ] 工程构建速度是否满足进阶用户需求？

---

### Step 4 · 角色C测试（边缘案例）

**用户画像**：输入不完整/矛盾/超出预期的内容，测试防错护栏

**测试场景矩阵：**

| 测试ID | 异常输入 | 期望护栏响应 |
|---|---|---|
| C-01 | 空输入/「开发Skill」无更多内容 | 输出追问清单 |
| C-02 | 要求「5分钟完成全部10阶段」 | 提示时间估算，建议分轮推进 |
| C-03 | 同时提供两个矛盾的材料 | 请用户确认优先处理哪个 |
| C-04 | 材料语言混杂（中文+英文+代码） | 正确处理多语言，不截断代码 |
| C-05 | 要求生成的SKILL.md包含违禁内容 | 拒绝并说明原因 |
| C-06 | 在阶段07要求「回到阶段02修改约束」 | 允许回溯，记录变更，重新门控 |
| C-07 | 用户连续输入「继续」「继续」而无实际内容 | 请求具体指令，不盲目推进 |
| C-08 | 要求生成超过10000字符的SKILL.md | 提示格式限制，提供精简方案 |

### Step 5 · 递归自举测试（Bootstrapping Self-Test）

**测试场景**：用「Skill Dev SOP」这个Skill来开发「Skill Dev SOP」自身

这是最严格的测试——如果一个工作流SOP的Skill能够正确指导自身的开发，证明其逻辑完全自洽。

**测试流程：**

```
Step 1: 触发词「我有一个SOP文档，想把它做成Skill」
  → 检测：激活是否正常？

Step 2: 提供「Skill开发工作流SOP v2.1.0.md」作为原始材料
  → 检测：价值分析是否识别出这是SOP自身？
  → 检测：是否产生循环引用/无限递归？

Step 3: 进入阶段01价值深挖
  → 检测：价值地图是否准确反映SOP的核心价值？

Step 4: 进入阶段02方向决策
  → 检测：设计约束书是否与实际SKILL.md一致？

Step 5: 进入阶段03工程构建
  → 检测：生成的SKILL.md是否与原始SKILL.md高度相符？
  → 检测：版本号处理是否正确（不应覆盖已存在的文件）？

Step 6: 递归场景处理
  → 检测：若用户要求「用生成的Skill来继续开发这个Skill」
  → 期望：识别递归场景，提示循环风险，建议跳出递归
```

**自举测试通过标准**：
- [ ] 无无限递归或循环引用
- [ ] 价值分析准确
- [ ] 生成的设计约束书符合SKILL.md实际约束
- [ ] 递归场景有明确警告

---

### Step 6 · 缺陷分级与汇总

测试完成后，将所有发现的问题按 P0/P1/P2 分级汇总：

**P0 · 必须立即修复（阻断性缺陷）**
> 导致Skill无法正常使用、格式规范违反、逻辑错误

**P1 · 本版本修复（重要缺陷）**
> 影响用户体验，但不阻断核心功能

**P2 · 下版本计划（改进项）**
> 优化建议，不影响当前版本发布

**缺陷报告模板：**

```markdown
# 缺陷报告 v1.0.0

## P0 · 立即修复（[N]项）

| ID | 发现场景 | 问题描述 | 修复方向 |
|---|---|---|---|
| P0-01 | [场景] | [问题] | [方向] |

## P1 · 本版本修复（[N]项）

| ID | 发现场景 | 问题描述 | 修复方向 |
|---|---|---|---|
| P1-01 | [场景] | [问题] | [方向] |

## P2 · 下版本计划（[N]项）

| ID | 发现场景 | 问题描述 | 规划版本 |
|---|---|---|---|
| P2-01 | [场景] | [问题] | v1.1.0 |

---
🔄 **自动进入阶段05 自主迭代修复**（共 P0:[N] P1:[N] P2:[N] 项缺陷）
```

## 防错护栏 | Error Guards

| 错误场景 | 护栏动作 |
|---|---|
| 测试脚本本身逻辑错误 | 重新审视测试设计，确保测试场景合理 |
| 所有测试通过（0缺陷） | 加入更严格的边缘测试，不轻易判定为0缺陷 |
| 递归测试产生实际无限循环 | 记录P0缺陷，立即终止递归，进入阶段05修复 |
| 缺陷数量过多（>20项） | 按P0→P1→P2优先级，仅将P0列入v1.0.0修复计划 |
