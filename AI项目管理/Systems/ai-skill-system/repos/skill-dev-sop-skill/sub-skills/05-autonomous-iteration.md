# Sub-Skill 05 · 自主迭代 Autonomous Iteration
## Skill Dev SOP — 阶段05 · v1.0.0

---

## 职责声明 | Responsibility

本子Skill接收阶段04的缺陷清单，**自主执行修复**，产出 v1.1.0 工程包与变更日志。修复遵循「三不原则」，优先级严格按 P0 → P1 推进，P2 仅记录入路线图。

**输入**：缺陷清单（P0/P1/P2分级）+ v1.0.0 工程包
**输出**：v1.1.0 工程包 + changelog 条目 + 修复验证报告
**门控**：汇报修复结果后继续（向用户展示修复摘要，等待 go/stop 确认）

---

## 三不原则 | Three Prohibitions

在执行任何修复前，必须遵循「三不原则」：

| 原则 | 内容 | 违反后果 |
|---|---|---|
| **不过度修复** | 只修复缺陷清单内的问题，不顺手修改「感觉不好」的内容 | 引入新缺陷，版本不可追溯 |
| **不静默修改** | 所有修改均记录在changelog，用户可审查每一条变更 | 用户失去控制感，信任崩溃 |
| **不跳过版本升级** | 修复后版本号必须升级（MINOR+1 for P0/P1，PATCH+1 for P2），不跳版本 | 版本历史混乱 |

## 执行逻辑 | Execution Logic

### Step 1 · 修复优先级排序

读取缺陷清单，按以下顺序排序修复序列：

```
修复执行序列
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
轮次1：所有 P0 缺陷（格式破坏/逻辑错误/功能阻断）
  → 必须全部修复，无一遗漏
  → 每项修复完成后立即验证

轮次2：所有 P1 缺陷（用户体验/引导改善）
  → 根据影响范围排序（影响多个阶段的优先）
  → 修复后检查是否引入新问题

轮次3：P2 缺陷处理
  → 不执行修复，仅将其写入路线图（changelog.md 的 Roadmap 段落）
  → 标注计划版本（v1.2.0 或更高）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

### Step 2 · P0 缺陷修复规程

对每一项 P0 缺陷执行精确修复：

**修复流程：**

```
1. 定位问题文件和位置（文件名 + 行号范围）
2. 分析根因（不只是症状）
3. 制定最小修改方案（避免过度修复）
4. 执行修改（Read → Edit，不跳过Read）
5. 立即验证修复效果（重新执行对应测试场景）
6. 记录变更到 changelog
```

**P0 常见修复场景：**

| 缺陷类型 | 根因 | 最小修复方案 |
|---|---|---|
| SKILL.md description 超1024字符 | 描述过于详细 | 删减冗余词，保留触发短语和核心功能 |
| 嵌套代码块格式错误 | 在```内嵌套``` | 改为段落描述或用 > 引用块替代 |
| 触发词未生效 | 触发短语缺失或格式错误 | 补充正确格式的触发短语 |
| 阶段门控逻辑错误 | 跳跃条件设置不当 | 修正门控条件，确保顺序推进 |
| 循环依赖（递归自举测试） | 接口设计循环 | 重构接口，引入中间转换层 |

---

### Step 3 · P1 缺陷修复规程

P1 修复优先考虑**用户体验**和**引导质量**：

**P1 常见修复场景：**

| 缺陷类型 | 修复方向 |
|---|---|
| 新手用户在某阶段卡住 | 补充引导话术 + 示例输入 |
| 错误提示不够友好 | 改写护栏响应，加入具体建议 |
| 阶段进展提示不清晰 | 添加进度标识（阶段N/10）|
| 某触发词未被识别 | 扩充触发短语列表（NAD原则） |
| 输出格式不一致 | 统一Markdown格式模板 |

---

### Step 4 · NAD 原则应用（触发短语优化）

在修复触发词相关问题时，应用 NAD 原则设计触发短语集：

| 原则 | 全称 | 在触发短语设计中的应用 |
|---|---|---|
| **N**atural | 自然表达 | 使用用户会自然说出的语言，不是命令式的技术指令 |
| **A**ccurate | 准确映射 | 每个触发短语都能准确对应Skill的核心功能 |
| **D**iverse | 多样覆盖 | 中英双语，正式/口语/命令式等多种表达方式 |

**触发短语质量检查清单：**
- [ ] 至少包含3个中文触发短语
- [ ] 至少包含2个英文触发短语
- [ ] 包含口语表达（不只是技术词汇）
- [ ] 无重复/近义词冗余
- [ ] 每个触发短语独立有效（不依赖组合）

### Step 5 · 版本号升级

所有修复完成后，执行语义化版本升级：

```
版本升级规则（语义化版本 MAJOR.MINOR.PATCH）

有 P0 修复 → v1.0.0 → v1.1.0（MINOR +1）
只有 P1 修复 → v1.0.0 → v1.0.1（PATCH +1）
只有 P2（路线图）→ 版本不变，仅记录

本阶段默认升级：
  有P0 → v1.0.0 → v1.1.0
```

## 防错护栏 | Error Guards

| 错误场景 | 护栏动作 |
|---|---|
| 修复后引入新P0缺陷 | 立即回滚，重新分析根因 |
| 用户要求修改P2（路线图项） | 提示：「P2项计划在下版本修复，当前版本聚焦P0/P1，确认跳出顺序吗？」 |
| 修复后描述character数超限 | 重新精简，不妥协格式规范 |
| 用户不同意某项修复方案 | 接受用户决定，但在changelog记录「用户决定不修复P[N]-[N]」 |
| P0数量过多导致修复轮次过长 | 按影响范围分批修复，每批完成后给用户中间确认 |
