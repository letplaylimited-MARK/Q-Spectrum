# Sub-Skill 06 · 使用指南 User Guide
## Skill Dev SOP — 阶段06 · v1.0.0

---

## 职责声明 | Responsibility

本子Skill生成面向**最终用户**（非开发者）的完整使用指南，采用「三层漏斗结构」设计信息架构，确保任何背景的用户都能找到适合自己的入口并顺利上手。

**输入**：v1.1.0 工程包（含全部已修复的核心文件）
**输出**：完整用户使用指南 `docs/user-guide.md`（含场景矩阵、步骤说明、FAQ、快速参考卡）
**门控**：用户确认场景覆盖完整后，进入阶段07通用封装

---

## 三层漏斗结构 | Three-Layer Funnel Architecture

使用指南采用「三层漏斗结构」设计，让不同背景的用户从自己的入口进入：

```
第一层：用户画像导航（我是谁？）
    ↓
第二层：场景选择（我要做什么？）
    ↓
第三层：步骤执行（我怎么做？）
```

这种结构避免了「一刀切」的文档问题——高手不需要从头读，新手不会迷路。

### Step 3 · 每个场景的详细步骤（第三层）

**场景A · 从零构想完整示例**

```markdown
### 场景A：我有一个想法，想做成Skill

**适用情况**：你有一个想法但不知道从哪里开始

**第一步：触发Skill**
> 说：「开发Skill」或「我想把[你的想法]做成Skill」

**第二步：描述你的想法**
Skill会问你几个问题，帮你把想法梳理清楚：
- 你想解决什么问题？
- 谁会用这个Skill？
- 现在没有这个Skill的话，你怎么做这件事？

**第三步：确认价值地图**
Skill会生成一份「价值地图」，展示你的想法的价值和开发方向。
→ 如果准确，回复「确认A/B/C」选择开发方向
→ 如果有偏差，直接告诉Skill哪里不对

**第四步：确认设计约束**
Skill会请你确认4个核心设计约束（C1-C4）
→ 这是整个Skill的「工程蓝图」，建议认真审阅

**第五步：等待工程构建**
确认约束后，Skill会自动生成完整的工程包（文件结构+核心文件）
→ 全程约3~5轮对话

**预期产出**：
- SKILL.md（可直接安装的官方格式文件）
- 三档系统提示词（完整版/精简版/激活卡）
- README.md + 使用指南
```

**场景B · 材料转化完整示例**

```markdown
### 场景B：我有现成材料，想做成Skill

**适用情况**：你有文档、SOP、脚本、代码，想包装成可复用的AI Skill

**第一步：触发Skill**
> 说：「我有[文档/脚本/SOP]要做成Skill」

**第二步：提供材料**
直接把材料发给Skill（支持Markdown/纯文本/代码/URL链接）
→ 材料越完整，价值分析越精准

**第三步：审阅价值地图**
Skill会从5个维度分析你的材料，找出最适合Skill化的部分
→ 特别关注「工程化潜力」评分和「差异化优势」

**第四步：快速通过阶段01~02**
有完整材料的用户通常可以快速确认方向和约束
→ 若材料已很完整，可以告知Skill「直接进工程构建」

**第五步：生成并审阅工程包**
Skill生成工程包后，重点检查：
- SKILL.md 的 description 是否准确描述了你的材料核心
- 触发短语是否覆盖用户实际会说的话

**预期产出**：
- 基于你材料的完整Skill工程包
- 可直接发布到GitHub并安装
```

### Step 4 · 常见问题 FAQ

```markdown
## ❓ 常见问题 FAQ

**Q: SKILL.md 的 description 限制1024字符是指什么？**
A: 字符数（len()），不是字数。一个汉字算1个字符，一个英文字母算1个字符，一个标点算1个字符。Skill会自动计算并在超限时提示。

**Q: 安装Skill需要GitHub账号吗？**
A: npx全局安装需要GitHub账号发布仓库，但如果只在本地使用，直接把SKILL.md粘贴到对应目录即可（方案A：本地文件）。

**Q: 我可以在中途改变方向吗？**
A: 可以。Skill支持在任何阶段回退并修改设计约束，但建议记录变更原因（在changelog中）。阶段越靠后修改代价越大。

**Q: 开发一个Skill需要多长时间？**
A: 典型场景：
- 简单Skill（单一功能）：5~8轮对话
- 中等Skill（3~5个子功能）：10~15轮对话
- 复杂SOP型Skill（完整工作流）：20~30轮对话

**Q: 如果Skill在模拟测试中发现了很多问题怎么办？**
A: 这是正常的。SOP的设计就是让问题在测试阶段暴露，而不是在用户使用时才发现。Skill会自主修复P0/P1缺陷，你只需要在修复后确认即可。

**Q: 发布到GitHub后，如何安装到CodeBuddy？**
A: 使用命令：`npx skills add [你的GitHub用户名]/[仓库名] --global --yes`
安装后输入触发词，Skill会自动激活。
```

## 防错护栏 | Error Guards

| 错误场景 | 护栏动作 |
|---|---|
| 使用指南中出现技术行话（YAML/frontmatter等） | 改为通俗说法，或加注解释 |
| 场景覆盖不完整（遗漏某用户类型） | 重新梳理C3目标用户，补充对应场景 |
| FAQ问题过于技术化 | 从用户视角重写，聚焦「我遇到了什么问题」 |
| 使用指南超过3000字 | 检查是否有冗余，优先保留「第三层步骤执行」内容 |
