# Sub-Skill 02 · 方向决策 Direction Decision
## Skill Dev SOP — 阶段02 · v1.0.0

---

## 职责声明 | Responsibility

本子Skill接收阶段01的价值地图与用户确认的方向，输出**设计约束书 Design Contract**——一份对Skill的4项核心约束进行精确定义的工程文档，作为整个开发过程的北极星。

**输入**：价值地图 + 用户确认的开发方向
**输出**：设计约束书 Design Contract（4项约束 C1~C4）
**门控**：用户逐项确认4项约束，全部确认后进入阶段03

---

## 执行逻辑 | Execution Logic

### Step 1 · 读取阶段01产出

从阶段01输出中提取：
- 确认的开发方向（A/B/C）
- Skill候选名称
- 核心价值摘要
- 优先级最高的功能模块
- 需要补足的局限性缺口

**若阶段01产出不完整**，先补完价值地图，再进入本阶段。

---

### Step 2 · 生成设计约束书 Design Contract

设计约束书包含4项核心约束，每项有明确的定义标准：

---

**C1 · 交付形态约束（Deliverable Form Constraint）**

定义Skill最终交付给用户的形态：

| 约束项 | 选项 | 本Skill选择 | 理由 |
|---|---|---|---|
| 主交付物 | SKILL.md / 系统提示词 / 工程包 / 三者组合 | [选择] | [理由] |
| 支撑文档 | 使用指南 / 子Skill模块 / 案例库 | [选择] | [理由] |
| 安装方式 | GitHub npx安装 / 直接粘贴 / 两者均支持 | [选择] | [理由] |
| 版本管理 | 语义化版本 MAJOR.MINOR.PATCH | ✅ 强制 | SOP标准 |

**C1约束声明示例：**
> 「本Skill交付完整工程包：SKILL.md（官方格式）+ 核心系统提示词（完整版+精简版+极简激活卡）+ 使用指南 + GitHub仓库，支持npx全局安装与直接粘贴两种安装方式。」

---

**C2 · 兼容性约束（Compatibility Constraint）**

定义Skill在哪些环境中必须能正常运行：

| 兼容维度 | 要求 | 实现方式 |
|---|---|---|
| 目标平台 | CodeBuddy / ChatGPT / Claude / 通义 / 文心 / 星火 / Gemini | 零平台专属语法 |
| 语言支持 | 中文主力 + 英文兼容 | 双语关键术语 |
| 模型能力要求 | 无特殊要求（模型无关设计） | 避免依赖特定API |
| 工具依赖 | Read/Write/Edit/Bash（最小权限原则） | 按需声明 |
| 输出格式 | Markdown结构化 | 统一输出模板 |

**C2约束声明示例：**
> 「本Skill采用模型无关设计，兼容全主流AI平台。核心功能不依赖工具调用，enhanced功能（文件读写）通过allowed-tools最小权限声明。」

---

**C3 · 目标用户约束（Target User Constraint）**

精确定义Skill服务的用户群体，避免「为所有人服务」的陷阱：

```
目标用户画像
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
主要用户：   [画像描述，含技术背景/使用场景]
次要用户：   [画像描述]
非目标用户：  [明确排除的用户类型，防止功能蔓延]

用户技术假设：
  □ 熟悉Markdown语法
  □ 了解AI提示词基础
  □ 有GitHub账号（可选）
  □ 了解Skill安装流程

用户痛点（Top 3）：
  1. [最大痛点]
  2. [次要痛点]
  3. [潜在痛点]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

**C4 · 成功标准约束（Success Criteria Constraint）**

定义Skill「做成了」的验收标准，分三层：

| 层级 | 标准 | 验证方式 |
|---|---|---|
| **最低成功标准（MVP）** | 用户能通过触发词激活Skill并完成1次完整流程 | 自举式模拟测试通过 |
| **预期成功标准** | 3类不同用户（新手/进阶/专家）均能顺利使用 | 3角色虚拟测试全通过 |
| **超出预期标准** | 用户无需阅读文档，依靠Skill内置引导即可完成全流程 | 零文档盲测通过 |

**关键性能指标 KPI（Skill专属）：**

```
核心KPI：
  触发响应时间：     ≤2轮对话（激活→开始执行）
  阶段推进清晰度：   用户始终知道自己在哪个阶段
  错误恢复能力：     遇到意外输入，护栏介入并引导恢复

质量KPI：
  SKILL.md格式合规：  description ≤1024字符
  门控完整性：        10个阶段，10个门控，无跳跃
  自举测试通过率：    ≥3/3 虚拟用户场景通过
```

---

### Step 3 · 约束书汇总与确认

将4项约束整合为完整的设计约束书，逐项展示给用户：

```markdown
# 设计约束书 Design Contract v1.0

## C1 · 交付形态
[C1约束声明]

## C2 · 兼容性
[C2约束声明]

## C3 · 目标用户
[C3用户画像]

## C4 · 成功标准
[C4 KPI表格]

---
⚠️ 请逐项确认：
□ C1 交付形态 — 我确认
□ C2 兼容性 — 我确认
□ C3 目标用户 — 我确认
□ C4 成功标准 — 我确认

全部确认后，我将启动阶段03工程构建。
```

**门控机制**：用户必须对每项约束明确确认（或提出修改），不得静默跳过。

---

### Step 4 · 约束冲突检测

在生成约束书后，自动检测以下常见冲突：

| 冲突类型 | 检测规则 | 处理方式 |
|---|---|---|
| C1 vs C3 冲突 | 「复杂交付形态」vs「非技术用户」 | 提示简化建议 |
| C2 vs C4 冲突 | 「强工具依赖」vs「全平台兼容」 | 分层功能设计建议 |
| C3 vs C4 冲突 | 「广泛用户群」vs「极高KPI标准」 | 建议用户群聚焦 |
| 约束过于模糊 | 成功标准无法量化验证 | 追问具体指标 |

---

## 防错护栏 | Error Guards

| 错误场景 | 护栏动作 |
|---|---|
| 用户跳过01直接进入02 | 回溯到01，完成价值分析再继续 |
| 用户对约束「全部默认」 | 提示：「设计约束书是工程保险，建议你至少对C3（目标用户）进行核实，避免后期返工」 |
| 约束项目过于宽泛 | 提问追问，直到约束可量化验证 |
| 用户要求修改已确认的约束 | 允许修改，记录变更原因，必要时重新触发门控 |
