# 设计原理文档 Design Principles
## skill-dev-sop · v1.0.0

---

## 一、自举式闭环工程哲学

「Skill 开发工作流 SOP」的诞生本身就是对其核心理念的最好证明：

> **用「超级提示词工程师」Skill 加工原始材料（Skill开发SOP文档），按照SOP的10阶段流程，生产出「Skill Dev SOP」Skill本身。**

这是一个完整的自举式闭环（Bootstrapping Loop）：

```
原始材料（SOP文档）
    ↓  [超级提示词工程师加工]
系统提示词工程
    ↓  [10阶段SOP流程]
Skill工程包
    ↓  [可以用该Skill开发下一个Skill]
新的原始材料
    ↑  [形成闭环]
```

这意味着：**任何知识体系，只要经过此Skill的加工，都可以成为新的AI Skill**。

---

## 二、三大核心原则详解

### IOSE — 子Skill拆解原则

**I · Independent（独立性）**

每个子Skill必须能够单独调用，不强依赖其他子Skill的执行结果。这使得：
- 用户可以从任意阶段进入（如「从阶段03继续」）
- 部分故障不影响整体（某阶段出问题，不影响其他阶段）
- 未来可以单独升级特定阶段

**O · One-responsibility（单一职责）**

每个子Skill只负责SOP的1个阶段，不跨阶段处理。这避免了：
- 职责混乱导致的输出不一致
- 过长的对话上下文影响质量
- 难以追溯问题根因

**S · Structured output（结构化输出）**

每个子Skill的输出必须遵循固定模板，包含：
- 执行摘要（本阶段做了什么）
- 核心产出（可复用的工程产物）
- 质量自检（关键指标验证结果）
- 下一步（门控条件 + 行动指引）

**E · Explicit trigger（明确触发）**

每个子Skill有明确的「进入条件」和「退出条件」：
- 进入条件：上游阶段的门控状态 + 用户确认信号
- 退出条件：本阶段产出完整 + 门控验证通过
- 无模糊地带：避免「好像完成了」的不确定感

---

### PAST — 安装提示词设计原则

**P · Paste-ready（即粘即用）**

激活卡片无需任何预处理，复制粘贴即可工作：
- 无需填写变量
- 无需调整格式
- 无需了解背景知识

**A · Activation signal（激活信号）**

激活卡片包含明确的自我声明：「当用户输入[X]时，我是[Y]，我会做[Z]」。这让AI模型能够：
- 清楚识别自己的角色
- 知道何时「开始工作」
- 避免在无关对话中意外激活

**S · Self-contained（自给自足）**

激活卡片包含所有必要的上下文，不依赖外部文件或历史对话：
- 核心触发词列表
- 激活后的初始输出
- 1~3条最关键的执行原则

**T · Trigger mapping（触发映射）**

激活卡片包含5+个不同表达的触发短语，覆盖：
- 中文/英文
- 命令式/口语式/描述式
- 不同详细程度的表达

---

### NAD — 触发短语设计原则

**N · Natural（自然表达）**

触发短语应该是用户「自然会说」的话，而不是需要记忆的命令：
- ❌ 命令式专有名词：「/skill-dev-sop activate」
- ✅ 自然语言：「我想把这个文档做成AI技能」

**A · Accurate（准确映射）**

每个触发短语必须准确对应Skill的核心功能：
- 触发词触发的行为 = 用户期望发生的事情
- 避免「触发了但做了不相关的事情」

**D · Diverse（多样覆盖）**

触发短语集合必须覆盖：
- 不同语言（中/英）
- 不同语气（正式/口语）
- 不同详细程度（「开发Skill」vs「我有一份Markdown文档想包装成可以在ChatGPT里用的AI技能」）

## 三、阶段门控设计哲学

**为什么需要阶段门控？**

在AI Skill开发实践中，最常见的失败原因不是「不会做」，而是「方向错了还在努力做」。阶段门控的设计目标是：

> 在每个关键决策点强制停下来，让用户确认「这是我要的方向吗？」

**门控类型分类：**

| 门控类型 | 触发条件 | 适用阶段 |
|---|---|---|
| 用户确认门控 | 等待用户明确表示「确认/继续」| 阶段01/02/06/09 |
| 自动推进门控 | 满足完成标准后自动进入下一阶段 | 阶段04→05 |
| 格式验证门控 | 通过自动化检查（字符数/格式规范）| 阶段07 |
| 测试通过门控 | 所有测试场景通过 | 阶段10 |

**三不原则与门控的关系：**

阶段门控实质上是「不过度修复」「不静默修改」「不跳过版本升级」三不原则的执行机制。门控确保了：
- 用户知道每个变更
- 每次推进都是有意识的决定
- 版本升级有迹可查

## 四、模型无关设计

**为什么要模型无关？**

AI模型生态发展迅速，专属某一平台的Skill会随平台更新而失效。模型无关设计的Skill：
- 今天在 CodeBuddy 上工作
- 明天在 ChatGPT 上同样有效
- 后天在新出现的AI平台上无需修改

**实现模型无关的5条规则：**

1. **零平台专属语法**：不使用 `<tool_use>` / `{{variable}}` / `[INST]` 等平台标记
2. **纯 Markdown 格式**：所有输出使用标准 Markdown，不依赖特定渲染引擎
3. **描述式工具声明**：`allowed-tools` 是建议而非强制，无工具时 Skill 降级运行
4. **语言自适应**：通过「自动检测用户语言」而非硬编码语言
5. **上下文最小化**：避免依赖超长历史上下文，每次响应自洽完整

## 五、自举式测试设计哲学

**什么是自举式测试（Bootstrapping Test）？**

> 用Skill A来测试Skill A自己的核心功能，形成自我验证闭环。

对于「Skill Dev SOP」，最强的自举测试是：

**「用 Skill Dev SOP 来开发 Skill Dev SOP 自身」**

如果一个「Skill开发工作流」的Skill，连指导自身的开发都做不到，那它的工作流就存在逻辑漏洞。

**自举测试通过标准：**
1. 无无限递归（识别自我引用，优雅处理）
2. 价值分析准确反映SOP的本质价值
3. 生成的设计约束书与实际SKILL.md约束一致
4. 生成的工程包结构与现有工程包高度相符

## 六、三层漏斗架构

使用指南的「三层漏斗结构」来自信息架构的核心原则：

```
第一层（最宽）：用户画像导航
  用户问：「我是谁？」→ 找到自己的入口
  
第二层（中间）：场景选择
  用户问：「我要做什么？」→ 选择对应场景

第三层（最窄）：步骤执行
  用户问：「我怎么做？」→ 按步骤操作
```

**设计依据**：
- 高手不需要从第一层开始（跳过导航直接到场景/步骤）
- 新手不会在第一层迷路（画像分类帮助定位）
- 所有用户都能在第三层找到精确操作步骤
