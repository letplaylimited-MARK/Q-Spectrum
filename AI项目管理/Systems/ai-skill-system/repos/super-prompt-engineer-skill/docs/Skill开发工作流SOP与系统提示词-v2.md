> 每个阶段包含：原始提示词分析、用户引导设计原则、阶段门控逻辑，以及可直接复用的标准模板。

### 阶段 01 · 价值深挖

#### 用户原始提示词解析

> 「请先充分理解并且深度思考此文档未来的价值应用」

| 要素 | 内容 | 效果评估 |
|---|---|---|
| 指令动词 | 充分理解 + 深度思考 | ✅ 要求全面分析，不只摘要 |
| 时间视角 | 未来的价值 | ✅ 锁定前瞻性分析维度 |
| 结构自由度 | 未指定输出格式 | ✅ 给 AI 自由组织框架的空间 |
| 缺少的要素 | 未指定分析维度数量 | ⚠️ 可能导致分析深度不一致 |

#### 用户引导设计原则

**核心目标**：让用户和 AI 在「这个 Skill 值得做、以及为什么值得做」上达成共识，**而不是立即进入构建**。跳过这步会导致后期频繁返工。

**AI 主动引导话术：**

```
我已阅读完您提供的材料。在开始构建之前，我想先和您确认价值判断：

📊 初步分析发现以下3个核心价值点：
1. [价值点A]——可解决 [用户群体] 的 [具体问题]
2. [价值点B]——可将 [手动操作] 转化为 [自动化流程]
3. [价值点C]——可在 [场景] 中节省 [时间/成本]

最有潜力的 Skill 化方向是：[方向描述]

请问您最认同哪个价值点？或者有我没提到的期待？
（您的回答将决定后续的设计重心）
```

#### 标准提示词模板（可复用）

```
请对我提供的【文档/资料/想法】进行以下深度分析，
分析完成后用选择题向我确认最核心的价值方向：

1. 核心价值识别：该内容解决了什么本质问题？（从用户痛点出发）
2. 应用场景矩阵：按用户类型（个人/企业/开发者）× 使用频率列表
3. 工程化潜力评分：哪些部分最适合 Skill 化？（1-10分，说明理由）
4. 局限性评估：现有版本的3个最大缺口
5. 竞争对比：现有同类工具/方案，我们的差异化优势是什么？

分析结束后，提出2-3个 Skill 化方向选项，用
[A/B/C 请选择] 的格式请我确认后再进入下一阶段。
```

### 阶段 02 · 方向决策

#### 用户原始提示词解析

> 「你有觉得该如何将这样的价值形成 Skill?」
> 「我希望这一个 Skill 是可以通用 AI 大模型的设计」

**深层解读**：用户采用了「半开放式决策对话」——提出方向意图，但保留决策权。第二句话是**约束植入**，一句自然语言确立了整个工程的核心设计原则（模型无关性）。

#### 设计约束书（Design Contract）模板

```markdown
## 设计约束书 · Design Contract
**Skill 名称**：___________
**版本**：v0.1（草案）
**确认日期**：___________

### 约束 C1：交付形态
- 选项：□ 单文件 SKILL.md  □ 完整工程包  □ 平台插件  □ 混合型
- 已确认：___________
- 理由：___________

### 约束 C2：兼容性边界
- 必须支持的模型/平台：___________
- 不需要支持的模型/平台：___________
- 语言策略：□ 中文  □ 英文  □ 中英双语自适应

### 约束 C3：目标用户（首要 → 次要）
- 第一优先级：___________（描述画像）
- 第二优先级：___________
- 第三优先级：___________

### 约束 C4：成功标准（可量化）
- 功能标准：用户能在 ___ 分钟内完成 ___ 操作
- 质量标准：输出稳定性 ≥ ___% 一致性
- 传播标准：其他用户能零修改直接使用

### 待决策项（进入阶段03前必须解决）
- [ ] ___________

> 用户确认：「我确认以上约束，可以进入工程构建阶段」
```

#### AI 主动引导话术

```
好的，基于价值分析，我建议将其开发为 Skill 工程包。动手之前，
需要锁定4项关键设计约束，这将决定整个工程的走向：

约束1：交付形态
A. 单文件 SKILL.md（简单，5分钟可用，功能有限）
B. 完整工程包（全面，支持迭代演进，首次开发推荐）
C. 平台插件（适合需要深度集成的技术场景）

约束2：兼容性
A. 专为 CodeBuddy 优化（功能最完整，但平台锁定）
B. 通用 AI 大模型（零门槛可移植，推荐）
C. 双轨并行（两个版本，工作量×1.5）

请逐一告诉我您的选择，或者说「全部按你建议的来」，
我生成设计约束书，您确认后开始构建。
```

### 阶段 03 · 工程构建

#### 核心引导原则：先结构，后内容

直接输出所有文件内容 = 信息淹没用户。正确做法：**先给文件结构树 → 用户确认 → 再逐一展开**。

#### 文件结构树确认模板

```
以下是基于设计约束规划的工程包结构：

[skill-name]/
├── SKILL.md                      ← CodeBuddy 官方格式入口（核心）
├── skill.md                      ← 通用平台说明（兼容备用）
├── core/
│   ├── system-prompt-full.md     ← 完整版系统提示词（~4000 tokens）
│   ├── system-prompt-lite.md     ← 精简版（~1500 tokens，轻量模型用）
│   └── activation-card.md        ← 极简激活卡片（即粘即用，≤200字）
├── sub-skills/
│   ├── 01-[能力A].md
│   ├── 02-[能力B].md
│   └── ...（共 N 个子模块）
├── deploy/
│   └── deploy-universal.md       ← 多场景部署指南
├── docs/
│   ├── changelog.md
│   ├── design-principles.md
│   └── simulated-usage-report.md
├── README.md
└── LICENSE

共 [N] 个文件，预计 [X] KB

请确认：
A. 结构合理，开始逐一生成文件内容
B. 我想调整：[具体说明]
C. 减少文件数量，只要核心文件
```

#### 子 Skill 原子化拆解——IOSE 原则

| 原则 | 含义 | 自检问题 |
|---|---|---|
| **I — Independent** | 可独立使用，无需其他子模块配合 | 单独粘贴这个 Skill，用户能完成完整任务吗？ |
| **O — One Responsibility** | 每个子 Skill 只做一件事 | 这个 Skill 的名字能用一个动词短语描述吗？ |
| **S — Structured Output** | 每次执行有确定性的输出格式 | 不同用户、不同输入，输出格式是否一致？ |
| **E — Explicit Trigger** | 触发词明确，用户不需要猜 | 用户说什么话会触发这个 Skill？ |

### 阶段 04 · 模拟测试（自举式调试）

#### 成本对比

| 测试方式 | 时间成本 | 金钱成本 | 发现问题质量 |
|---|---|---|---|
| 等真实用户反馈 | 数周~数月 | 可能已造成口碑损失 | 取决于用户表达能力 |
| 组织用户访谈 | 3-5天 | 中等 | 较高，但样本有限 |
| **AI 自举式模拟（本方案）** | **30-60分钟** | **零成本** | **高，且可精确定位** |

#### 三类极端用户画像

```
用户A · 极小白（代表最脆弱的用户）
├── 技术背景：从未写过代码，第一次用 AI
├── 耐心阈值：遇到一次困惑就放弃
├── 表达方式：口语化、不完整、不专业
└── 核心测试：Skill 能否在不预设专业知识的情况下完成任务

用户B · 典型目标用户（代表最普遍的使用场景）
├── 技术背景：会用 ChatGPT，但不懂提示词工程
├── 目标：解决工作中的具体问题，不想学原理
├── 表达方式：带具体业务背景，有明确结果期望
└── 核心测试：Skill 能否在「用户不理解原理」时输出专业结果

用户C · 挑剔的专业用户（代表最苛刻的标准）
├── 技术背景：API 用户，懂提示词原理，会评估质量
├── 诉求：效率优先，输出要可程序化处理
├── 表达方式：简洁精准，直接要结果
└── 核心测试：Skill 能否满足专业用户的质量和效率标准
```

#### 模拟测试标准提示词（可复用）

```
请以以下3类用户的真实视角，对 [Skill名称] 进行完整虚拟使用测试。

要求：
- 逐步输出「用户输入 → AI 回复 → 用户体验评价」的完整对话
- 每当用户遇到困惑、卡壳、失望时，用 ⚠️ 标记并说明原因
- 测试结束后汇总问题，按以下标准分级：
  P0：用户无法完成核心任务（必须在本版本修复）
  P1：用户体验受损但可绕过（本版本修复）
  P2：优化建议（列入下个版本计划）

[用户A] 技术背景：___  目标：___  开始语句：「___」
[用户B] 技术背景：___  目标：___  开始语句：「___」
[用户C] 技术背景：___  目标：___  开始语句：「___」

测试完成后，不等指令，立即修复全部 P0 和 P1 级问题。
```

### 阶段 05 · 自主迭代

#### 迭代执行的「三不原则」

1. **不过度修复**：P2 级问题不在本阶段处理，列入 changelog 待办区。
2. **不静默修改**：每次修改必须记录「修改前 → 修改后 + 修改原因」。
3. **不跳过版本升级**：即使只修复一个 bug，也必须升级版本号（PATCH +1）。

#### 迭代记录标准格式

```markdown
## 变更记录 v[X.X.X] · [日期]

**变更触发**：[模拟测试/用户反馈/主动优化]
**修复级别**：P0×[N] / P1×[N]

### Fix #[编号] · [问题标题]（P[级别]）
- **受影响文件**：`[文件路径]`
- **问题描述**：[用户在哪个场景遇到了什么问题]
- **根本原因**：[为什么会出现这个问题]
- **修改前**：`[原始内容片段]`
- **修改后**：`[修改后内容片段]`
- **验证方式**：[如何确认修复有效]

### 待修复（v[X.Y.Z] 计划）
- [ ] #[编号] [问题标题]（P2）
```

### 阶段 06 · 使用指南

#### 用户引导「三层漏斗」结构

```
第一层：用户画像导航（30秒定位自己）
├── 我是对话用户（直接粘贴使用）→ 跳至第2章
├── 我是进阶用户（配置平台）→ 跳至第3章
├── 我是开发者（API集成）→ 跳至第4章
└── 我是企业管理员（团队部署）→ 跳至第5章

第二层：场景选择（2分钟找到对应方案）
├── 我用 ChatGPT/Claude → 2.1
├── 我用文心/通义/豆包 → 2.2
├── 我用 CodeBuddy/Cursor → 2.3
└── 我用本地模型/API → 2.4

第三层：步骤执行（5分钟完成接入）
└── 步骤 + 代码示例 + 成功验证 + 常见问题
```

#### 使用指南生成提示词（可复用）

```
请为 [Skill名称] 生成完整使用指南，遵循「三层漏斗」结构：

第一层：用户画像快速导航表（30秒定位章节）

第二层：按场景分章节，至少覆盖：
- 主流对话平台（ChatGPT/Claude/文心/通义/豆包）
- 系统提示词配置（平台设置中持久激活）
- API 调用（Python + Node.js + curl 示例）
- 本地模型（Ollama / LM Studio）
- 工作流平台（n8n / Dify / Coze）
- IDE AI 工具（CodeBuddy / Cursor / Continue）
```

### 阶段 09 · 安装提示词生成

#### PAST 原则

| 原则 | 含义 | 失败案例 |
|---|---|---|
| **P — Paste-ready** | 粘贴即用，无需修改占位符 | 含 `[your_api_key]` |
| **A — Activation signal** | 有明确的激活确认信号 | AI 激活后沉默，用户不确定是否生效 |
| **S — Self-contained** | 完全自包含，不依赖外部上下文 | 「如上文所说的角色定义」（跨会话失效） |
| **T — Trigger mapping** | 能力与触发词映射清晰 | 用户不知道说什么话能触发哪个功能 |

### 阶段 10 · 账户级全局安装

#### 全局安装工作原理

```
npx skills add [repo] --global --yes
    │
    ├─ 1. 从 GitHub clone 仓库到本地临时目录
    ├─ 2. 识别仓库根目录的 SKILL.md 文件
    ├─ 3. 安装到 ~/.agents/skills/[skill-name]/
    ├─ 4. 为支持的 Agent 创建 symlink
    │     ├─ CodeBuddy → ~/.codebuddy/skills/[skill-name]
    │     ├─ Cursor   → ~/.cursor/skills/[skill-name]
    │     └─ ...其他支持的 Agent
    └─ 5. 触发短语注册到 Agent 的 Skill 索引
```

**关键认知**：全局 Skill 是「**按需激活**」而非「常驻激活」——收到触发词才加载，避免每个会话都消耗 Context 窗口。

#### 触发短语设计——NAD 原则

| 原则 | 含义 | 示例 |
|---|---|---|
| **N — Natural** | 用户会自然说出的话 | ✅「帮我优化这个提示词」 ❌「启动提示词工程师模式」 |
| **A — Accurate** | 不会被其他能力误触发 | ✅「设计工作流」 ❌「帮我」（过于宽泛） |
| **D — Diverse** | 覆盖同一意图的多种表达 | ✅「优化提示词/改善指令/这个 prompt 跑偏了」 |

## 第四部分：关键提示词工程洞察

### 5 类高效提示词模式

**模式一：价值优先型**（阶段01）

> 「请先充分理解并且深度思考此文档未来的价值应用」

关键词是「**先**」（阻止 AI 急于产出）和「**深度**」（触发维度扩展而非浅层摘要）。先建立「Why」共识，再进入「How」构建，省略这步会导致整个工程方向偏移。

**模式二：约束锁定型**（阶段02）

> 「我希望这一个 Skill 是可以通用 AI 大模型的设计」

一句自然语言确立了整个工程最核心的约束。**约束型提示词不需要专业术语，意图清晰即可**——「通用 AI 大模型」虽不是精确技术描述，但 AI 能准确识别到「不能用平台专属语法」的工程含义。

**模式三：角色扮演测试型**（阶段04）

> 「你有办法自己模拟有用户使用了 skill 之后，该如何自主优化吗?」

整个项目**投入产出比最高**的一句提示词。触发了「自举式调试」——AI 既是开发者又是用户，在同一会话内完成构建→测试→迭代的完整闭环。零成本发现10项真实缺陷。

**模式四：维度扩展型**（阶段06）

> 「...等等（需深度思考更多维度）」

「等等（需深度思考更多维度）」这个短语明确告诉 AI：你的列举是起点而非边界，触发主动延伸。没有这个短语，AI 往往只覆盖用户列出的内容，遗漏重要场景。

**模式五：类比路径型**（阶段10）

> 「比如现在有 Agent/Plan...再新增这一个技能」

提供「类比参照物」让 AI 准确理解目标状态。当用户描述 AI 未必直接知道答案的目标时，**提供一个类比物比精确描述技术路径更有效**——触发了文件系统探索→发现 npx skills CLI 的完整路径。

### 三个关键决策点的深度复盘

| 决策点 | 面临的分叉 | 实际选择 | 长期价值 |
|---|---|---|---|
| 阶段02：交付形态 | 单文件 vs 完整工程包 | 完整工程包 | 模块化结构使迭代可精准修改单文件；单文件方案无法支持局部迭代 |
| 阶段04：测试方式 | 等真实用户 vs AI自主模拟 | AI自主模拟 | 零成本发现10项缺陷；AI模拟能主动构造极端场景，真实用户只反馈常规体验 |
| 阶段10：安装方式 | 每次粘贴 vs 全局安装 | 全局安装 | 触发短语永久激活，传播成本趋近于零；每次粘贴导致实际使用频率大幅下降 |

### 四条沟通效率核心规律

**规律一：「门控」比「催促」更有效**

设置阶段门控虽看起来更慢，但大幅减少返工——方向偏差在早期被截住，而不是在最终交付时才发现。

**规律二：「选择题」比「开放问题」高效 3-5 倍**

开放问题要求用户从零开始思考所有可能性；选择题只需判断哪个更符合直觉。当需要用户决策时，始终提供 A/B/C 选项。

**规律三：「先结构，后内容」避免信息淹没**

大型工程输出策略：先给目录/结构树（让用户掌握全貌）→ 获确认 → 再逐一展开（让用户聚焦局部）。直接输出全部内容无法有效审查质量。

**规律四：「约束」越早锁定，返工越少**

阶段02锁定的4项约束为后续8个阶段所有细节决策提供参照框架。每遇小分叉，对照约束书即可快速决策，无需回到用户层面重新确认。

### 常见问题与解决方案

| 问题现象 | 根本原因 | 解决方案 |
|---|---|---|
| SKILL.md 验证失败 | description 超过1024字符 | 精简描述，触发短语移到正文，用字符计数器验证 |
| 全局安装后 Skill 不触发 | 触发短语不够自然 | 回顾NAD原则，覆盖更多口语表达方式 |
| AI 输出不稳定/幻觉 | 提示词缺乏确定性约束 | 增加：「基于用户提供信息，不虚构；确定性任务温度=0」 |
| 精简版效果远差于完整版 | 压缩时丢失了关键约束 | 精简版必须保留：角色定义+触发词映射+最小执行原则 |
| GitHub 推送 403 | OAuth Token 过期 | 重新执行 `source get_token.sh github` 获取新 Token |
| 子 Skill 相互依赖 | 违反 IOSE 原则的「I」 | 提取公共依赖为「共享上下文」模块，各子 Skill 独立引用 |
| 模拟测试问题发现太少 | 测试画像不够极端 | 用「第一次用AI的60岁用户」和「只看结果的CTO」替代普通用户 |
| 用户觉得输出太多难以审查 | 未遵循「先结构后内容」 | 大量输出前先给目录，询问「是否展开全部内容」 |
