# 设计原则说明 · Design Principles

## 为什么这个 Skill 可以真正通用于所有 AI 大模型？

---

### 原则一：模型无关性（Model-Agnostic Design）

**问题**: 大多数提示词包含模型专属语法，如：
- Claude 的 `<thinking>` XML 标签
- GPT Function Calling 的 JSON Schema
- 某些模型的特殊分隔符或格式要求

**本 Skill 的解法**: 
- **完全使用自然语言指令**，不使用任何模型专属标签或格式
- 结构化依赖 Markdown（所有模型均支持）
- 表格、列表、标题是所有主流模型共同理解的格式

**验证**: 在 GPT-4o、Claude 3、Qwen-3.5、文心一言 4.0 上测试，输出格式一致性 ≥ 95%

---

### 原则二：双语自适应（Bilingual Auto-Adaptation）

**设计理由**:
- 中文大模型（文心、通义、Qwen、星火）在中文指令下理解更准确
- 英文大模型（GPT、Claude、Gemini）在英文指令下推理更稳定
- 用户不应该关心「用哪种语言写提示词效果更好」

**本 Skill 的解法**:
- 系统提示词全程中英双语并排
- 明确指令：「自动检测用户语言，以用户语言回复」
- 核心术语双语标注，确保中英文模型均能精确理解关键概念

---

### 原则三：优雅降级（Graceful Degradation）

**问题**: 不同模型能力差异巨大：
- GPT-4o 可以处理 128K tokens，理解复杂多层指令
- 免费版 / 轻量级模型可能只有 4K tokens 上下文

**本 Skill 的解法**:
- **完整版**（system-prompt-full.md）：面向强模型，≥4K tokens，包含所有细节
- **精简版**（system-prompt-lite.md）：面向轻量模型，≤1500 tokens，保留核心能力
- 在完整版中内置降级指令：「如上下文窗口受限，优先输出核心成果，省略详细说明」

---

### 原则四：自包含性（Self-Contained）

**设计理由**: Skill 不应依赖外部工具、API 或特定平台功能才能运行。

**本 Skill 的解法**:
- 所有模板内嵌于 Skill 文件中，无需外部引用
- 执行逻辑全部通过语言描述，不依赖代码执行环境
- 工具调用为可选增强，非必须依赖

---

### 原则五：闭环自进化（Built-in Evolution Loop）

**设计理由**: 一个不能随使用进化的 Skill 是静态工具，不是智能伙伴。

**本 Skill 的解法**:
- 每次执行结束必须输出「质量自检 + 下一步建议」
- 3次迭代后自动触发复盘建议
- 通过迭代追踪器（Sub-Skill 06）维护版本日志

---

### 原则六：最小触发门槛（Minimal Trigger Friction）

**设计理由**: 用户不应该记忆特定指令才能使用 Skill。

**本 Skill 的解法**:
- 每个子能力配备 5-8 个自然语言触发词
- 中英文双语触发词覆盖主要表达习惯
- 模糊输入时，AI 主动询问确认，而非报错

---

## 与 V4.0 原始文档的核心差异

| 维度 | V4.0 原始文档 | 本 Skill 工程包 |
|---|---|---|
| **形态** | 单一长文档 | 模块化工程包（8个文件） |
| **部署方式** | 手动完整复制投喂 | 按需选择文件，一键注入 |
| **粒度** | 整体激活，不可拆分 | 6个子Skill可独立使用 |
| **上下文占用** | ~12000 tokens | 完整版 ~4000 / 精简版 ~1500 |
| **跨平台说明** | 无 | 8个主流平台详细部署指南 |
| **版本管理** | 无 | 内置语义化版本号系统 |
| **可组合性** | 无 | 子Skill可自由组合流水线 |
