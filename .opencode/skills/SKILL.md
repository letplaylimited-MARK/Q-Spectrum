# 智腦工作流 / Brain Workflow Skill

## 啟動順序（Session Startup）

```
每次會話嚴格依此順序執行：
  0. 呼吸 0.5s — 確認自己在哪個專案
  1. 健康檢查 — 執行 verify-integration.py，逐項確認不跳過
  2. 知識回顧 — 讀取 _HANDOFF/STATUS.md 知道當前階段
                → 讀取 _HANDOFF/CRITICAL-REMINDERS.md 知道 P0 問題
                → 讀取 _HANDOFF/MEMORY-INDEX.md 回顧已結晶知識
  3. 任務確認 — 輸出任務確認單給使用者
  4. 三級強度選擇 — Tier 1(輕量) / Tier 2(標準) / Tier 3(全面)
  5. 執行 — 邊做邊記錄、邊檢查、邊修復
  6. 評審 — 雙軌迭代（開發軌 + 評審軌）
  7. 記憶沉澱 — 提取 P0-P1 知識，更新記憶系統
  8. 會話結束確認 — 總結完成/未完成/待辦
```

## 業務工作流（B-001）

```
[修復] → [規劃] → [重構] → [精簡] → [驗證] → [沉澱]
   ↑                                      │
   └────────── [評審 FAIL] ←──────────────┘

每節點必須產出一個 diff 或變更記錄。
修復優先：已知 bug 先修再開發新功能。
```

## 數據工作流（D-001）

```
[感知] → [解析] → [決策] → [執行] → [評估] → [記憶]
   │        │        │        │        │        │
   ├─用戶輸入     ├─角色路由  ├─檔案操作   │        │
   ├─LSP 診斷    ├─風險評估  ├─SQL 查詢   │        │
   ├─知識檢索    ├─任務分解  ├─Shell 執行 │        │
   └─全域搜索    └─架構審查  └─Git 操作   │        │
                                          │        │
           ╰────────── 知識庫 ────────────╯        │
           ╰────────── 長期記憶 ──────────────────╯
```

## 觸發工作流（T-001）

| 觸發事件 | 動作 |
|---------|------|
| 會話啟動 | §1 啟動順序（記憶確認 + 健康檢查） |
| 發現 bug | 立即修復，不論當前階段 |
| 測試失敗 | 暫停當前工作，先修測試 |
| 文件與程式不一致 | Q06 審查優先 |
| 使用者說「緊急」 | T01 介入，跳過非關鍵步驟 |
| 3 視角不一致 | 增加第 4 視角（T01 裁決） |

## 調用工作流（C-001）

```
使用者
  │
  v
Orchestrator（本體）
  │
  ├── Task tool ──→ 開發軌 Subagent (Q01/Q03/Q04)
  │                  └── read/write/edit/bash/grep/glob
  ├── Task tool ──→ 評審軌 Subagent (Q06/S01/S02)
  │                  └── read/grep（唯讀）
  └── Task tool ──→ 協調 Subagent (T03)
                     └── read + 沙盤推演

規則：
  1. Orchestrator 只分解任務，不直接操作工具
  2. 評審軌禁止 write/edit/bash（唯讀審查）
  3. 禁止循環調用
```

## 評審檢查表

```
□ 安全問題？           → 評審軌有覆蓋權
□ 數據不一致？         → 評審軌勝
□ 架構爭議？           → 升 T01 裁決
□ 風格/偏好？          → 開發軌勝
□ 評審給替代方案了沒？  → 沒給就不算有效評審
```

## 記憶寫入閘門

```
寫入條件（全符合才寫）：
  □ 可驗證的事實（非猜測）
  □ P0-P1 級（非一次性資訊）
  □ 新的知識（非重複）
  □ 有來源/證據（非無根據）
```
